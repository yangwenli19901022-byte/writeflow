# WriteFlow 产品需求文档（PRD）

> 项目代号：WriteFlow  
> 文档版本：v1.0  
> 目标：定义 MVP 阶段产品范围、功能需求与验收标准，作为设计与开发的唯一需求来源。

---

## 1. 产品概述

WriteFlow（产品中文名暂定为项目代号）是一款面向内容创作者的 **AI 辅助写稿工具**。用户输入原始素材（文字、采访记录、资料摘要等）和写作要求，AI 根据指定的体裁与风格生成可直接使用或稍作修改的文案。

### 1.1 产品定位

- **一句话描述**：让 AI 帮你把素材快速改写成不同体裁的文案。
- **核心价值**：降低内容创作门槛，缩短从素材到成稿的时间。
- **差异化**：专注中文写作场景，内置新闻、社媒、公众号等专业体裁模板，输出结构清晰、风格可控。

### 1.2 目标用户

| 用户类型 | 典型场景 | 核心诉求 |
|---------|---------|---------|
| 新媒体运营 | 把活动资料改写成小红书/公众号文案 | 快、有网感、符合平台调性 |
| 自媒体博主 | 根据录音转写或灵感片段生成文章 | 节省时间、保留个人风格 |
| 记者/编辑 | 根据采访素材写新闻消息或评论 | 客观、准确、结构规范 |
| 企业市场人员 | 把产品资料改写成演讲稿/推文 | 专业、有说服力 |

### 1.3 使用场景

1. 运营人员拿到一份产品资料，需要生成 5 条小红书种草文案。
2. 记者整理完采访素材，需要快速产出 800 字新闻消息。
3. 自媒体作者有一个故事梗概，需要扩展成一篇公众号长文。

## 2. 问题陈述

### 2.1 用户痛点

- **写作启动难**：面对零散素材，不知道如何组织成文章。
- **体裁切换成本高**：同一素材需要适配公众号、小红书、新闻等不同平台，每次都要重新改写。
- **AI 输出不稳定**：通用 Chat 工具生成的内容风格不一致，需要反复调教。
- **安全顾虑**：直接把 API Key 放在前端或本地工具中存在泄露风险。

### 2.2 我们要解决什么

- 提供 **按体裁封装的 AI 写作工作流**，用户只需选择体裁、填写要求，即可生成结构化的成稿。
- 通过服务端代理，确保 **API Key 安全**。
- 支持 **流式输出**，提升生成过程的体验。
- 保存 **历史记录**，方便用户回溯和二次编辑。

## 3. 用户故事

### 3.1 MVP 用户故事

作为新媒体运营，我希望粘贴产品资料后选择"小红书文案"，以便快速获得多条带 emoji 和标签的种草文案。  
作为记者，我希望粘贴采访素材后选择"新闻消息"，以便生成符合倒金字塔结构、突出 5W1H 的短新闻。  
作为公众号作者，我希望输入故事素材后选择"公众号文章"，以便获得一篇有标题、分段清晰、语言流畅的长文。  
作为所有用户，我希望看到 AI 逐字生成内容，以便在等待时有更好的反馈感。  
作为所有用户，我希望保存每次生成的结果，以便后续查看或重新编辑生成。  

### 3.2 二期用户故事（不在 MVP）

- 支持上传图片、音频转写。
- 支持自定义 Prompt 模板。
- 支持导出 Word / PDF。
- 支持用户登录和云端同步。

## 4. 功能需求

### 4.1 功能总览

| 模块 | 功能点 | MVP 优先级 |
|-----|--------|-----------|
| 素材输入 | 多行文本输入、字数统计 | P0 |
| 写作要求 | 体裁选择、风格/调性、字数、补充要求 | P0 |
| AI 生成 | 生成、停止、重新生成、流式输出 | P0 |
| 结果展示 | 复制全文、复制标题、Markdown 预览 | P0 |
| 历史记录 | 保存草稿与生成结果、左侧列表、继续编辑 | P1 |
| 提示词模板 | 内置 3 个体裁 Prompt 模板 | P0 |
| 设置 | 模型选择（Kimi / 通义 / Claude）、API Key 配置 | P1 |

### 4.2 详细功能说明

#### 4.2.1 素材输入区

- 输入框支持多行文本，最小高度 200px，可拖拽扩展。
- 实时显示当前素材字数。
- 输入内容为空时，生成按钮禁用或提示"请输入素材"。
- 支持占位文案提示用户可输入什么（如：采访记录、会议纪要、产品资料、故事梗概等）。

#### 4.2.2 写作要求区

**体裁选择（单选）**：
- 公众号文章
- 小红书文案
- 新闻消息

**目标读者 / 调性（单选，默认"通用"）**：
- 专业严谨
- 轻松口语
- 煽情走心
- 犀利观点
- 通用

**字数要求（下拉/输入）**：
- 短（300 字以内）
- 中（300-800 字）
- 长（800-1500 字）
- 自定义（输入框）

**补充要求（多行文本，可选）**：
- 用户可以填写特殊要求，如"突出性价比"、"不要出现价格"、"模仿某公众号风格"等。

#### 4.2.3 AI 生成区

- 点击"生成"后，按钮变为"生成中..."并显示 loading。
- 内容以流式方式逐字/逐段显示。
- 生成过程中可点击"停止"。
- 生成完成后可点击"重新生成"（使用相同素材和要求）。
- 结果区支持复制全文、复制标题。
- 结果以 Markdown 格式渲染（标题、列表、加粗等）。

#### 4.2.4 历史记录

- 左侧任务列表显示所有草稿。
- 每个任务显示标题（可自动生成前 20 字）和最后更新时间。
- 点击任务可加载素材和要求到编辑区。
- 支持删除历史任务。
- MVP 阶段使用数据库（Supabase / Vercel Postgres）持久化存储。

#### 4.2.5 提示词模板

- 每个体裁对应一个独立的 Prompt 模板文件。
- 模板结构包含：角色定义、体裁规范、素材、要求、输出格式。
- 服务端根据体裁动态组装 Prompt 并调用大模型。

#### 4.2.6 设置

- 默认模型为 **Kimi（Moonshot AI）**。
- **用户需在自己的设置页面填写 Kimi API Key**，应用通过后端代理调用 Kimi API。
- API Key 仅存储在用户浏览器（localStorage/sessionStorage），后端不持久化存储，仅作为请求代理使用。

## 5. 非功能需求

### 5.1 性能

- 首屏加载时间 < 3 秒（在 4G 网络下）。
- AI 生成首字响应时间 < 5 秒（正常网络下）。
- 流式输出无明显卡顿。

### 5.2 安全

- 用户填写的 API Key 通过后端代理调用 Kimi API，禁止前端直接请求第三方 API。
- 后端不持久化存储用户 API Key，仅用于当前请求转发。
- 前端存储 API Key 时建议使用 sessionStorage 或加密后的 localStorage。
- 用户输入需做基本的安全过滤，防止 Prompt 注入。
- 生成内容建议用户人工审核后再发布。

### 5.3 可用性

- 界面响应式，支持桌面端和移动端访问。
- 主要操作路径不超过 3 步：输入素材 → 选择体裁 → 点击生成。
- 关键操作有明确反馈（loading、成功提示、错误提示）。

### 5.4 可扩展性

- 新体裁的添加只需新增一个 Prompt 模板文件和配置项。
- 前端组件化，便于后续扩展更多输入类型（图片、音频）。

## 6. 信息架构与页面结构

### 6.1 页面结构

```
/                         # 首页（写作工作台）
/api/generate             # 服务端 API：AI 生成接口
```

### 6.2 首页布局

桌面端采用 **三栏布局**：

```
+------------------------------------------+
|  顶部导航栏：Logo + 设置按钮               |
+----------+------------------+------------+
|          |                  |            |
|  任务    |    素材输入区    |   生成结果  |
|  列表    |   + 写作要求区   |    展示区   |
|          |                  |            |
+----------+------------------+------------+
```

移动端折叠为 **步骤式流程**：
1. 输入素材
2. 选择体裁和要求
3. 查看结果

### 6.3 关键组件

- `AppHeader`：顶部导航
- `TaskSidebar`：左侧任务列表
- `InputPanel`：素材输入 + 写作要求表单
- `OutputPanel`：生成结果展示 + 操作按钮
- `GenreSelector`：体裁选择器
- `ToneSelector`：调性选择器
- `GenerateButton`：生成/停止按钮

## 7. 用户流程

### 7.1 核心流程：生成文案

```
打开应用
  ↓
输入/粘贴原始素材
  ↓
选择体裁（公众号/小红书/新闻消息）
  ↓
设置调性、字数、补充要求
  ↓
点击"生成"
  ↓
服务端组装 Prompt → 调用大模型 API
  ↓
流式返回结果，前端逐字展示
  ↓
用户复制结果或重新生成
  ↓
自动保存草稿与生成历史
```

### 7.2 历史流程

```
点击左侧历史任务
  ↓
加载对应素材和要求
  ↓
用户可修改后再次生成
```

## 8. 数据需求

### 8.1 数据实体

#### Draft（草稿/任务）

| 字段 | 类型 | 说明 |
|-----|------|------|
| id | string | 唯一标识 |
| title | string | 任务标题，取素材前 20 字 |
| material | text | 原始素材 |
| genre | enum | 体裁：公众号/小红书/新闻消息 |
| requirements | json | 调性、字数、补充要求等 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

#### Generation（生成结果）

| 字段 | 类型 | 说明 |
|-----|------|------|
| id | string | 唯一标识 |
| draft_id | string | 关联草稿 |
| model | string | 使用的模型 |
| prompt_version | string | Prompt 模板版本 |
| output | text | 生成文本 |
| tokens_used | int | 消耗 Token 数（可选） |
| created_at | timestamp | 创建时间 |

### 8.2 存储策略

- **MVP**：使用 **Supabase** 或 **Vercel Postgres** 保存草稿与生成结果，支持跨设备同步和数据持久化。
- **后续**：根据用户量考虑迁移到更独立的数据库服务。

## 9. AI / 提示词需求

### 9.1 模型选择

- 默认使用 **Kimi（Moonshot AI）**（中文长文本处理能力强、上下文窗口大）。
- 后续可扩展支持 **DeepSeek-V3 / 通义千问 / Claude 3.5 Sonnet**。

### 9.2 提示词模板规范

每个体裁独立维护一个 Prompt 模板文件，包含：

1. **角色定义**：明确 AI 扮演的写作专家身份。
2. **体裁规范**：说明该体裁的结构、语言、风格要求。
3. **素材注入**：用户输入的原始素材。
4. **要求注入**：用户选择的调性、字数、补充要求。
5. **输出格式**：规定标题、正文、标签等输出结构。

### 9.3 输出格式示例（小红书文案）

```markdown
# 标题
[吸引眼球的标题]

# 正文
[emoji 丰富、口语化的种草文案]

# 标签
#标签1 #标签2 #标签3
```

### 9.4 质量要求

- 公众号文章：结构完整（标题、引言、正文、结尾），语言流畅，无明显 AI 腔。
- 小红书文案：使用 emoji，口语化，有种草感，带 3-5 个相关标签。
- 新闻消息：采用倒金字塔结构，包含 5W1H 要素，语言客观简洁。

## 10. 成功指标

### 10.1 北极星指标

- **单用户周生成次数**：衡量产品核心价值是否被认可。

### 10.2 过程指标

- 生成成功率（API 调用成功比例）> 95%
- 平均生成首字响应时间 < 5 秒
- 用户重新生成率（衡量输出质量）< 30%
- 结果复制率 > 60%

### 10.3 定性指标

- 用户反馈生成内容是否可直接使用或仅需少量修改。
- 用户是否愿意推荐给他人。

## 12. 风险与假设

### 12.1 风险

1. **生成质量不达标**：不同模型对中文写作的理解有差异，需预留 Prompt 调优时间。
2. **API 成本超支**：流式输出和长文本生成消耗 Token 较多，需设置限制。
3. **内容合规风险**：AI 可能生成不实或敏感内容，需提示用户审核。

### 12.2 假设

1. 目标用户主要使用桌面端进行长文本创作。
2. 用户愿意为高质量、省时的 AI 写作工具付费（二期验证）。
3. 第三方大模型 API 的可用性和稳定性在可接受范围内。

## 14. 附录

### 14.1 术语表

- **MVP**：Minimum Viable Product，最小可行产品。
- **SSE**：Server-Sent Events，服务器推送事件，用于实现流式输出。
- **Prompt**：发送给 AI 模型的指令文本。
- **Token**：大模型计费单位，通常 1 个中文汉字约等于 1-1.5 个 Token。
